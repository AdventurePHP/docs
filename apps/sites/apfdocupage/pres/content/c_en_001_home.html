Version 1.10 (stable) released!

The APF team is proud to anounce the stable release of the 1.10 series (04.09.2009). Within this maintenance release, development entirely switched to PHP5. A compatibility version for PHP4 will be available since the beginning of 2010. Beneath some clean-ups and refactorings , several performance optimization tasks guarantee excellent performance. The probably biggest technical innovation is the introduction of the "dependency injection container" similar to the objection initialization of SPRING (JAVA). Further, this release contains many features, that were discussed in the developer forum to be included.

Morover, the final release of the 1.10 branch contains all changes concerning the feedback on the two release candidates.


Introduction of PHP5 language elements:
As of this release, development completely switched to PHP 5. This means, that the code now contains all relevant PHP5 language elements. The PHP 4 version is only a generated compatibility version. For this reason, you must now take care about the visibility of methods and class members. Please ensure, that the visibility is recognized within your existing applications.

Refactoring of the APF filter api:
Due to errors with the PHP version >=5.2.10 and >=5.3.0, the filter API was refactored. This affects the signature of the filter(), that now does only expect one parameter - the input. Details can be taken from the the API documentation of the AbstractFilter class. These changes also affect the form filters. For further information, please consult the filter documentation.


Cleanups & optimization!
Along with the code cleanups, the following classes are renamed:
  • serviceManager -> ServiceManager
  • sessionManager -> SessionManager
  • benchmarkTimer -> BenchmarkTimer
Further, the comment module now uses the captcha module to secure the form and brings a much more flexible HTML layout, that can be adapted using CSS. Besides, the __extractTagLibTags() parser method was optimized for less CPU and RAM consumption. These changes affirm the performance tests described on http://adventure-php-framework.org/Seite/103-Yii-vs-APF.


Introduction of the DIServiceManager!
As of release 1.10, the DIServiceManager was introduced. This component provides a "dependency injection container", that enables the developer to initialize application components with configurable parameters as desired. In the past, dependency injection was only possible for well defined params within the framework - language and context of the current application. Please refer to chapter http://adventure-php-framework.org/Page/107-Services for a detailed description of the dependency injection implementation of the APF and the object initialization possible with the new component. The section also includes application samples.


Enhancement of existing components:
In the course of refactoring placed in the 1.10 branch, several components were reworked to meet the extended needs. Among these are the umgtManager, the GenericORMapper and the MySQLHandler. As discussed in the german language forum, the business component umgtManager now features some more loadUserBy*() methods. So ist is much more easy to select users by given keys within applications based on the usermanagement module. To easy statement debugging, the MySQLHandler now supports statement logging into a separate log file. Now you can see all statements sent to the database - especially generated ones - looking at the statement logfile. Based on this change, the GenericORMapper can be created in "debug mode" using an additional factory switch.


Feedback on the RCs:
The most frequently discussed topic was the compatibility to the PHP5.3 version. For this reason, the filter API and the SessionManager was adapted as well as some descendent components. Beneath, some errors within the iterator tag and the GenericORMapper were fixed and the functionality of the ServiceManager and the RequestHandler were improved.

Version 1.9 (stable) released!

Today we are proud to announce the availability of the release 1.9 (stable). This production release contains:

Usermanagement:
The usermanagement module was subjected to refactoring within the 1.9 series. This means:
  • More easy inclusion into existing applications via taglib.
  • Business component as a central interface to the module.
  • Enhanced documentation with many examples.
  • Direct usage of the user management database through the GenericORMapper.
Concerning these changes and by force of the circumstance, that is is built on top of the GenericORMapper, it can be uses as a reusable and multitenant user base for use within various applications.


Filter!
The filter issue was completely changed for the 1.9 series. The advantages of the new implementation are:
  • More easy configuration of the input- and output filter via the registry.
  • Transparent adaption of the filters to your application cases without the need to change existing functionalities.
  • Creation of various url layouts (e.g. developing a CMS).
  • Form filter to secure user input on text fields.

Remake of approved components:
In the course of refactoring placed in the 1.8 branch, several components were reworked to meet the extended needs. These are:
  • ImageManager: Refactoring eases usage of the class and provides a more convenient API.
  • Pager: Refactoring, that it can be used to page result sets loaded by the GenericORMapper. The new session caching facility gains performance of the module.

Global error and exception handling!
The error handling of the framework is now arranged in a more flexible way. This means:
  • The component, that is used by the framework to handle upcomming errors can now be configured by use of the registry. You can, too, define your own error handler, if necessaray.
  • Additionally, the PHP 5 release now contains a seperate exception handling. The mechnism is similar to the error handling included in the framework.
This mechanism enables the developer to catch unhandled exceptions within an application. The documentation now provides extensive information on the functionality, the extension and the configuration of this feature.


Cleanup!
As mentioned in the changelog of the 1.8 release, deprecated components have been removed. These are:
  • filesystemHandler
  • abstractCacheManager
  • cacheV4Manager
  • objectCacheManager
  • statCacheManager
  • bbCodeParser
The article Migration from 1.8 to 1.9 contains extended release notes as well as hints on the migration to the new framework version.


Feedback to the Release Candidate:
The stable release contains bugfixes of the baseControllers, taken from the user feedback. Further, the image delivery actions were optimized. Beneath, the documentation was enhanced and a new code highlighting tool was introduced.
  • The methods __getForm(), __templatePlaceholderExists(), __placeholderExists(), __getTemplate(), and setPlaceHolder() do no longer create errors on neighbour object access in the PHP5 release version.
  • The StreamMediaAction out of the tools::media::actions namespace and the ShowImageAction out of the modules::socialbookmark::biz::actions namespace now send an enhanced set of HTTP headers, that enable the browser so cache the images.
Details on the topics presented, can be read about in the chanelog on the downloads page. There, you can also find hints on the discussions on the 1.9-RC1 release within the forum.
